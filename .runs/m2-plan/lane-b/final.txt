```md
#### Lane B. CLI/Docs/Help Surface

Lane B owns the operator-facing `pipeline compile` contract cutover for the first supported M2 wedge. It does not own compile assembly logic; it owns the exact CLI shape, help posture, success/refusal/proof presentation rules, and the docs/help parity gates that make the new surface shippable.

CLI shape:

- Ship exactly two supported invocations:
  - `system pipeline compile --id <pipeline-id> --stage <stage-id>`
  - `system pipeline compile --id <pipeline-id> --stage <stage-id> --explain`
- `--id` reuses the published canonical-id and unambiguous shorthand rules from the existing `pipeline` family.
- `--stage` accepts a canonical stage id or an unambiguous shorthand stage id, but resolution stays scoped to the selected pipeline.
- `--explain` is a boolean proof-mode switch, not a separate command family.
- Do not add positional target syntax, raw-path selectors, hidden aliases, `--output`, `--format`, `--json`, runner/profile override flags, or any compile-time route mutation flags in M2.
- `pipeline compile` stays a thin `crates/cli` wrapper over one compiler-owned typed compile result; CLI glue must not become a second authority for compile targeting, payload assembly, or proof rendering.

Help/docs posture:

- Add `compile` under `system pipeline --help` only when compiler behavior, shared proof corpus, snapshots, and docs land together.
- Help text for `compile` should describe it as the explicit stage-payload surface, not as a generic workflow runner.
- Help text for `--explain` should say: render compile proof only; do not print payload.
- Keep the top-level product story unchanged: `pipeline` remains the orchestration surface, `generate` remains the packet surface, `inspect` remains the packet proof surface, and `doctor` remains the recovery surface.
- Document one explicit carveout: compile-specific proof lives on `pipeline compile --explain`; this does not broaden `inspect` into a generic compile-proof command.

Success/refusal posture:

- Plain `pipeline compile` success is payload-only stdout.
- Payload-only means: print only the compiled stage payload bytes, with no `OUTCOME`, no trust header, no route-basis recap, no success summary, and no trailing `NEXT SAFE ACTION`.
- `pipeline compile --explain` success is proof-only stdout.
- Proof-only means: do not print the compiled payload body again. Instead print a trust-header proof view that points back to plain `pipeline compile` when the operator wants the payload.
- Refusal output is shared across plain and explain modes and stays compact:
  - `OUTCOME: REFUSED`
  - `OBJECT: <pipeline-id>:<stage-id>`
  - `NEXT SAFE ACTION: <exact repair command>`
  - then one refusal block with category, summary, and broken subject
- Freshness, inactive-stage, selected-stage-not-in-pipeline, and missing-required-input refusals must remain distinct operator outcomes with distinct recovery guidance.

What stays out of `inspect`:

- No compile target selector or compile-specific flags on `inspect`.
- No compile proof sections inside packet proof output.
- No route-basis freshness dump for successful compile requests.
- No include-expansion ledger, required/optional input decision ledger, or payload preview for compile inside `inspect`.
- No “inspect the compiled stage” fallback that would turn `inspect` into a second compile surface.
- `inspect` remains the proof surface for packet composition and decision evidence only; `pipeline compile --explain` owns compile proof.

How the two modes should read:

- Payload-only mode should read like a copy/paste-ready stage payload, not like a receipt.
- Proof-only mode should read like an evidence review with stable ordering:
  - target
  - route basis used for freshness
  - selected stage metadata
  - include expansion
  - required vs optional input decisions
  - output contract
  - gating notes
  - payload summary only
- For `stage.10_feature_spec`, proof mode must explicitly show that runner/profile material came from the resolved route basis and from the stage includes, including `runners/${runner}.md`, `profiles/${profile}/profile.yaml`, `profiles/${profile}/commands.yaml`, and `profiles/${profile}/conventions.md`.
- Proof mode may summarize payload size or sections, but it must not duplicate the payload body that plain compile prints.

Implementation sequencing in `crates/cli` plus docs:

- First, require the compiler-owned compile interface and renderer contract to exist; Lane B should not recreate compile rendering ad hoc in `crates/cli/src/main.rs`.
- Then add `PipelineCommand::Compile` plus a dedicated `PipelineCompileArgs` shape with exactly `--id`, `--stage`, and `--explain`.
- Then add a thin `pipeline_compile` dispatcher in `crates/cli` that:
  - resolves repo root the same way as other `pipeline` commands
  - forwards canonical target selection to compiler-owned code
  - prints payload-only success for plain mode
  - prints proof-only success for `--explain`
  - preserves compiler-owned refusal classes and wording
- Then update CLI help snapshots and help-drift parity checks for:
  - `system --help`
  - `system pipeline --help`
  - `system pipeline compile --help`
- Then update docs/help surfaces in one cut:
  - `docs/SUPPORTED_COMMANDS.md`
  - `docs/CLI_PRODUCT_VOCABULARY.md`
  - `docs/CLI_TONE_RULES.md`
  - `docs/CLI_OUTPUT_ANATOMY.md`
  - `docs/CLI_COMMAND_HIERARCHY.md`
  - `docs/contracts/C-02-rust-workspace-and-cli-command-surface.md`
- Only after those docs and snapshots match should `pipeline compile` be described as supported in the M2 product story.

Lane B boundary note:

- The current shared foundation proof fixture still stops at `stage.07_foundation_pack`. M2 help/docs must not claim the `stage.10_feature_spec` compile wedge is shipped until the shared proof corpus is extended to cover that stage and both payload/proof goldens exist.
```
